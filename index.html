不管是露營愛好者，或是露營新手，你都有機會找到最適合自己的露營方式

專題介紹 Project Overview

  • 專題背景

    • 根據中華民國露營協會統計，台灣的露營人口早已超過200萬人，越來越多人對於這項活動產生興趣並嘗試體驗。然而琳瑯滿目的露營裝備與露營場地，往往讓未曾嘗試的露營麻瓜們對於這項活動只敢遠觀，而對於擁有資深經驗的露營達人們來說，如何快速且有系統性地找到自己理想中的營地也是一大挑戰。 我們透過建立整合露營資訊的平台，集結了露營商城、場地，以及各種相關知識內容，並提供線上交易及預訂服務，幫助露營愛好者省去至不同網站查找營區及設備的麻煩，讓人們在繁忙生活步調中，也能夠輕鬆找到最符合自己理想的露營方式。


  • 開發動機

    • 在這次專題開發中，個人負責的範圍為商城中購物車、結帳，和訂單管理的部分，希望可以藉由本次專案將課程中的內容學以致用，針對這次負責的功能，除了嘗試使用Redis建立購物車，在結帳和訂單成立的環節，為了能確保金額不會被使用者自前端竄改， 商城的結帳金額都自後端抓取商品資料庫中的金額資訊來去計算，在專題製作過程中，也不斷期許自己能夠以最貼近實務的方式去撰寫相關功能。


  • 技術使用

    • 此專案以Spring Boot實作RESTful API，將業務邏輯與資料存取分層管理，前端使用fetch發送HTTP請求串接API，實現前後端分離，後續會針對個人負責部分更詳細說明。


功能說明 Key Features

加入購物車

  • 不管登入與否皆可將商品加入購物車
  • 可透過商城首頁中展示的商品品項，選擇顏色及規格後點選加入購物車
  • 進入商品頁面後，也可在選擇顏色、規格、數量後點選加入購物車

編輯購物車內容

  • 針對顏色、規格、數量於購物車中進行編輯
  • 針對單筆商品從購物車中移除
  • 將商品購物車中所有商品清空

合併購物車

  • 在結帳前會先進行登入驗證，登入驗證完成後可將先前存放購物車內容與未登入情況下存放購物車內容合併

結帳

  • 填寫結帳基本資訊，包含結帳方式、出貨方式、收件人資訊
  • 可透過linepay進行付款
  • 設置出貨方式和付款方式的綁定限制，若選超商付款僅能使用超商取貨，反之亦然
  • 收件人資訊可一鍵帶入會員資訊

使用折價券

  • 結帳時提供未使用過且在使用期間之折價券供會員使用
  • 依據折價券設定最低使用金額限制結帳時可否使用
  • 若使用折價券，則會依據折價券設定的折扣金額或折扣比例進行折扣計算

申請取消、申請退貨、新增/編輯評論

  • 可依據訂單狀態不同去做申請取消或退貨的操作
    • 功能 條件限制
    申請退貨 1. 訂單狀態必須為3（已取貨）且 退貨申請狀態需為0（未申請退貨）
    取消訂單 1. 訂單狀態為0（未出貨）或1（待出貨）
    2. 只能在未出貨前申請取消
    新增/編輯評論 1. 訂單狀態為3（已取貨）且 退貨申請狀態需為0（未申請退貨）

後台管理＿編輯訂單

  • 可根據訂單狀態提供後台管理員不同編輯權限
    • 編輯項目 條件限制
    收件人基本資料 只在訂單狀態為待賣家確認時可編輯，如已進入出貨流程將無法更改
    出貨日期 已取貨完成或已取消禁止編輯
    折價券 在訂單還在待賣家確認狀態可隨時新增、修改，系統會即時計算金額，但訂單進入出貨環節時將無法操作
    退貨處理 僅限訂單狀態為已取貨完成且退貨申請狀態為申請退貨時管理員可針對退貨成功與否進行狀態更新

實作細節 Implementation Details

Redis 購物車快取管理與合併邏輯

在設計購物車時將存放方式依據登入會員與否分為兩種， 在未登入情況下會將資料存放在前端，一旦登入後就會先檢查資料庫是否有先前登入後存放到購物車的商品，並且將前後端的購物車進行合併， 合併過程會使用商品編號、商品顏色、商品規格來去確認該商品是否先前已存入資料庫，若曾經存入就調整數量；若未存入過就新增一筆資料。 

                  
  // 取得已登入會員的購物車（Redis）
  String redisKey = REDIS_KEY_PREFIX + memId; // Redis的key

  // 產生商品唯一key（商品ID:顏色ID:規格ID）
  String hashKey = guestItem.getProdId() + ":" + guestItem.getProdColorId() 
      + ":" + guestItem.getProdSpecId();

  // 取出購物車已存在的商品
  Object value = redisTemplate.opsForHash().get(redisKey, hashKey);
  CartVO existItem = null;
  
  if (existItem != null) {
    existItem.setCartProdQty(existItem.getCartProdQty() + guestItem.getCartProdQty());
    // 直接存 CartVO 物件
    redisTemplate.opsForHash().put(redisKey, hashKey, existItem);
  } else {
    // 購物車沒有該商品，須新增一個存在購物車中
    CartVO.CartKey cartKey = new CartVO.CartKey(memId, guestItem.getProdId(), 
      guestItem.getProdColorId(), guestItem.getProdSpecId());
    CartVO newVO = new CartVO();
    newVO.setCartKey(cartKey);
    newVO.setCartProdQty(guestItem.getCartProdQty());

    // 直接存 CartVO 物件
    redisTemplate.opsForHash().put(redisKey, hashKey, newVO);
  }

// 回傳合併後的購物車DTO清單給前端
return getCart(memId);


訂單金額與折扣運算邏輯

在新增訂單同時也要去新增訂單明細，以下程式碼為訂單新增訂單中新增明細的部分， 在取得明細同時才可以從商品表格和商品規格表格取得商品真正的售價，並且計算出折價前總金額後再來看折價券折價部分。 折價方式又分為兩種，一為總額減去折價金額，二為總額打折（百分比折扣），扣除折價金額並加上運費後才算是整筆訂單的實付金額。 

                  
  // 於明細建立完成後取得折扣前總金額，先預設為BigDecimal 0，方便後續計算訂單金額
  BigDecimal beforeDiscountAmount = BigDecimal.ZERO;
  sovo.setBeforeDiscountAmount(beforeDiscountAmount.intValue());

  // 將實付金額預設為 BigDecimal 0，方便後續計算訂單金額
  BigDecimal afterDiscountAmount = BigDecimal.ZERO;
  sovo.setAfterDiscountAmount(afterDiscountAmount.intValue());

  for (ShopOrderItemsDetailsDTO_insert_req detailsDTO : dtoDetails) {

      ShopOrderItemsDetailsVO soidVO = new ShopOrderItemsDetailsVO();

      // 取得數量，先轉為BigDecimal方便後續計算
      BigDecimal qty = new BigDecimal(detailsDTO.getShopOrderQty());
      soidVO.setShopOrderQty(qty.intValue());

      // 取得價格，先轉為BigDecimal方便計算
      ProdSpecListVO.CompositeDetail2 key = new ProdSpecListVO.CompositeDetail2(detailsDTO.getProdId(),
          detailsDTO.getProdSpecId());
      ProdSpecListVO spec = pslr.findById(key).orElseThrow(() -> new RuntimeException("查無該種規格"));

      // 商品折扣
      ShopProdVO prod = spr.findById(detailsDTO.getProdId())
          .orElseThrow(() -> new RuntimeException("查無該種商品"));
      ;
      BigDecimal prodDiscount = prod.getProdDiscount();

      BigDecimal price = new BigDecimal(spec.getProdSpecPrice()).multiply(prodDiscount).setScale(0,
          RoundingMode.HALF_UP);
      ;

      // 將價格存入訂單明細
      soidVO.setProdOrderPrice(price.intValue());

      // 依照明細計算折扣前金額
      beforeDiscountAmount = beforeDiscountAmount.add(price.multiply(qty));

    }

  // 取得discountCodeType
  if (sovo.getDiscountCodeId() != null) {
    Byte type = sovo.getDiscountCodeId().getDiscountType();

    // 取得discountValue
    BigDecimal value = dcVO.getDiscountValue();

    if (type == 0) {

      sovo.setDiscountAmount(value.intValue()); // 將value轉換為Integer

      // 計算折扣後金額
      BigDecimal totalAmount = beforeDiscountAmount.subtract(value);

      // 加入運費計算實付金額
      afterDiscountAmount = totalAmount.add(shipFee);

      // 將afterDiscountAmount存入商品訂單
      sovo.setAfterDiscountAmount(afterDiscountAmount.intValue());
    } else if (type == 1) {
      // 折扣前金額*(discountValue(%))並四捨五入
      BigDecimal discountAmount = beforeDiscountAmount.multiply(value).setScale(0, RoundingMode.HALF_UP);

      // 計算折扣後金額，先以BigDecimal計算
      BigDecimal totalAmount = beforeDiscountAmount.subtract(discountAmount);
      

      // 加入運費計算實付金額
      afterDiscountAmount = totalAmount.add(shipFee);

      sovo.setDiscountAmount(discountAmount.intValue()); // 存入折扣金額
      sovo.setAfterDiscountAmount(afterDiscountAmount.intValue()); // 存入實付金額

    }
  }


訂單狀態與異常流程控管

在訂單編輯的部分，提供管理員很大的彈性去做調整，但也希望能夠符合實務業務流程，所以依據訂單狀態、退貨申請狀態也設定許多編輯限制。 如後台管理中，有限制說一旦進入到退貨申請流程時，訂單就不在允許狀態變更，其餘欄位如也會有對應的限制， 如聯絡資訊、送貨地址也有限制在訂單未出貨的情況下才能進行編輯。 因訂單狀態也會涉及到是否可以進行訂單明細中評分、評價編輯，所以在這裡也有設計只有在完成取貨的狀況下才能進行評分、評價編輯。 

                  
// 訂單狀態變更，判斷是否可執行
if (dtoUpdate.getShopOrderStatus() != null
    && !sovo.getShopOrderStatus().equals(dtoUpdate.getShopOrderStatus())) {
    // 退貨申請流程開啟時，不允許直接變更狀態
    if (sovo.getShopReturnApply() == 0 || sovo.getShopOrderStatus() != 5) {
        sovo.setShopOrderStatus(dtoUpdate.getShopOrderStatus());
    } else {
        throw new IllegalArgumentException("商品訂單已進行退貨申請流程");
    }
}

// 只有「已取貨」且未申請退貨的訂單才能新增/修改商品評論
final int ORDER_STATUS_COMPLETED = 3, NO_RETURN_APPLY = 0;
if (order.getShopOrderStatus() != ORDER_STATUS_COMPLETED && order.getShopReturnApply() == NO_RETURN_APPLY) {
    throw new RuntimeException("訂單未完成，無法評論或修改評價");
}

心得與收穫 Reflection

在短短五個月中，從第一次 print 出 Hello World，到與組員合作完成專題開發，這段過程讓我深刻體會到自己在程式語言基礎知識與團隊合作方面，還有許多需要學習與成長的空間。雖然經過這次專題，我對相關內容有了更深入的了解，但也發現未來仍有許多可以改進的地方，期許期許自己能在之後的學習中持續精進，並將所學應用於實際專案開發中。

在時間與能力有限的情況下，這次專題開發仍有許多未盡完善之處。例如在購物車功能上，雖然已經使用 Redis 做快取，但資料一致性處理仍有更細緻的優化空間；在訂單管理部分，除了後端使用 DTO 與 @Pattern 進行資料驗證外，前端也設置了多層資料驗證並以 alert 提醒使用者。這讓我意識到如何避免前後端重工、簡化程式碼，是我未來需要深入研究的課題。

總結來說，雖然這次專題無法做到盡善盡美，但能將課程中所學實際應用於專案開發，對我而言已是莫大的收穫。未來我會持續努力學習，並期望能在日後的專案中持續成長與突破。
